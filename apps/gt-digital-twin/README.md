# GT Digital Twin · Engine Showcase v1

Internal-shareable, single-page dashboard for the **Lockport Energy Associates** CCGT engine.
Energy-only (no capacity, no steam). The page communicates engine mechanics and scenario spread — **not** asset economics.

---

## What this is

A static React/Vite frontend that reads two precomputed JSON files produced by the **real engine**:

- `web/client/public/precomputed.json` — knob-grid headline + sensitivity + waterfall + per-path forward (consumed by §02–§08)
- `web/client/public/monthly_forecast_panel.json` — real per-month forward panel consumed by §01's interactive 6-panel chart (generated by `scripts/build_monthly_forecast_panel.py`)

The engines:

- `gt_engine.run_path` — historical / aged-state replay
- `forward.run_forward` — 25-window probability-weighted forward, RT basis, seed 42

The dashboard never simulates anything. It reads from a 12-cell knob matrix:

| dimension          | values                              |
| ------------------ | ----------------------------------- |
| dispatch policy    | A · B · C  (ADR-009 bracketing)     |
| Henry Hub gas mult | 0.75× · 1.0× · 1.25×                |
| initial state      | aged (per-mode historical end-state) + fresh (EOH 24k) |

= 3·3 aged cells + 3·1 fresh cells = **12 cells**.

`run_forward` takes ~80–110 s per cell on this machine. Live recompute on every
slider move is not viable — precompute + key-hash lookup is.

## Run

### 1) Precompute (one-time per assumption change)

```bash
cd /home/user/workspace/GT_Modeling
python apps/gt-digital-twin/scripts/precompute.py
```

Writes to `apps/gt-digital-twin/web/client/public/precomputed.json` (~125 KB).
Total runtime ≈ 16–22 min (3 historical replays + 12 forward sweeps).

### 1b) Build the real-data monthly panel (one-time per assumption change)

```bash
cd /home/user/workspace/GT_Modeling
python apps/gt-digital-twin/scripts/build_monthly_forecast_panel.py
```

Writes to `apps/gt-digital-twin/web/client/public/monthly_forecast_panel.json`
(~125 KB). This is the **real-data** monthly export consumed by §01's
interactive 6-panel forecast chart — it carries the real per-month engine
outputs (generation, gross margin, fired hours, net P&L) plus per-month SEAS5
temperature, forward LMP, and Henry Hub gas drivers from the path inputs.

> The older `build_monthly_trajectories.py` is **DEPRECATED** (illustrative monthly
> shape) and will print a deprecation message and exit if invoked. Use
> `build_monthly_forecast_panel.py` instead.

### 2) Dev

```bash
cd apps/gt-digital-twin/web
npm install         # one-time
npm run dev         # http://localhost:5000
```

### 3) Build (static)

```bash
cd apps/gt-digital-twin/web
npm run build       # → dist/public
```

The output in `dist/public` is fully static (the only network read is
`/precomputed.json`). No backend at runtime.

## Constraints honored (verbatim from product brief)

- **Energy-only / not-valuation banner** is persistent on every screen.
- **Capacity + steam revenue** are deferred — visible as disabled roadmap controls
  in the Future Modules group, each with status badge + reason + upgrade path.
- **Active controls for v1**: scenario, A/B/C policy, gas price, initial-state preset.
- **Disabled roadmap controls**: capacity, steam, deeper LTSA, multi-year horizon,
  custom wear constants — all visible with `deferred` / `placeholder` / `assumed_industry`
  badges and tooltip explanations.
- **Per-assumption info button/popover** with status badge + value + source +
  why-acceptable-for-v1 + what-would-upgrade-it.
- **No tornado chart** — replaced with `What matters most` ranked cards as the user
  specified. Each card has a swing bar, rationale, status badge, and source.
- **Engine mechanics are NOT reimplemented in JS**. Every number on the page comes
  from `precomputed.json`, which was produced by the real Python engine. There is
  no fake hidden logic, no client-side simulation, no derived headline.
- **B≈C explanation** auto-shows when comparePolicies is on AND aged is selected
  (the only state where the B/C overlap is visible).
- **No default Streamlit purple, no Plotly defaults, no speedometers, no pie charts,
  no generic metric-card slop.** Custom warm-bone/charcoal palette derived from the
  Nexus design system, restrained Hydra teal accent, Instrument Serif for one
  editorial moment (the hero), IBM Plex Mono + tabular-nums for every number.

## Status taxonomy (from `docs/assumptions/parameter_calibration_register.md`)

Each control + roadmap item is tagged with one of:

| status                 | meaning                                                             | color    |
| ---------------------- | ------------------------------------------------------------------- | -------- |
| `real_observed`        | Direct observation (Henry Hub, scenario ensemble, etc.)             | green    |
| `assumed_industry`     | Tier-2 cited (GER-3620, Kumar 2012, ADR-009 dispatch shape)         | amber    |
| `modeling_convention` | Internal default to keep the demo legible (e.g. EOH=24k fresh init) | blue     |
| `placeholder`          | Structurally correct, magnitudes not contract-grade (LTSA terms)    | mauve    |
| `deferred`             | Excluded from v1 by design (capacity, steam, multi-year horizon)    | grey     |

## Design decisions

- **React/Vite over Streamlit.** Precomputed JSON makes the dashboard genuinely
  static-deployable; no Python runtime needed for the preview link. Streamlit's
  default visual language also conflicts with the "no slop" rule.
- **Pure SVG charts, no Recharts/Plotly.** Total visual control over the waterfall
  and distribution. No library default styling leaks through.
- **Custom palette** (warm bone `#F7F6F2` light / charcoal `#171614` dark, Hydra
  teal `#01696F` accent) — Nexus-aligned, derived from product brand cues, with
  status colors imported from the calibration register taxonomy.
- **Typography:** Geist for UI, Instrument Serif italic for one hero moment,
  IBM Plex Mono for every number with `tabular-nums`. Two type families, three
  weights, no decoration.
- **No localStorage / sessionStorage** anywhere (sandbox blocks it).
- **One persistent control bar on the left**; on mobile it stacks above the canvas.
  Sticky header + banner remain across scroll.

## What's live vs. precomputed

| element                  | source                                                                |
| ------------------------ | --------------------------------------------------------------------- |
| Headline P10/P50/P90 net | precomputed forward grid → `quantiles.net_pl_usd`                     |
| KPI cards                | precomputed forward grid → `quantiles.{spark, ltsa, mwh, fired_hours}` |
| Waterfall                | precomputed historical decomposition (`historical_decomp[mode]`)      |
| Distribution dot plot    | precomputed `per_path` arrays (3 modes overlaid when compare is on)   |
| What matters most cards  | precomputed sensitivity ranks (delta vs A · gas 1.0 · aged baseline)  |
| Assumption table         | precomputed `calibration_register` + roadmap statuses                 |
| Engine baseline panel    | precomputed `constants` + `aged_state_summary`                        |

Off-grid combinations (e.g. gas 0.9×) are not shown. The grid is the
declared scope; users can see exactly the cells that were simulated.

## Honest limitations

1. **Not a cockpit.** It is a precomputed-grid showcase, by design. `run_forward`
   takes ~110 s per cell — live recompute on slider move is not viable on this
   machine and would require infrastructure (warm Python worker, queue, etc.)
   that is out of scope for v1.
2. **1-year forward only.** The B≈C overlap is structural on 1 year. Multi-year
   is where C's headroom-bounded regime actually bites. That's why multi-year
   is a disabled roadmap control with the badge + explanation.
3. **Energy-only.** Cogen survivorship is largely capacity + steam revenue. The
   energy-only headline is deliberately **not** a valuation. The banner says so
   on every screen.
4. **LTSA terms are placeholders.** Athens-prototype values, structurally
   correct stream count and shape but not the Lockport contract.

## Path to a live cockpit (handoff)

The cleanest upgrade keeps the same frontend, replaces the JSON read:

1. Wrap `gt_engine.run_path` + `forward.run_forward` in a thin FastAPI service
   with a knob-hash cache (LRU keyed by `(policy, gas_mult, init_state)`).
2. Warm the cache at startup with the 12 baseline cells (current precompute).
3. Add a `POST /run` endpoint that returns from cache when present, runs the
   engine when not. Slider-move latency drops to ~50ms for warm cells, ~80–110s
   for cold cells (with a clear loading state).
4. Add a "save scenario" persistence layer when multiple users want to compare
   their explorations.

The frontend `lib/data.ts` already has the `keyFor()` helper that produces the
hash the server would use.
